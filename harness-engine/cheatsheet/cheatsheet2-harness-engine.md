# Harness Engine 一页速查表

> 基于苏格拉底测试（方法3）作答情况定制，重点标注测试中暴露的薄弱点。
> 测试时间：2026-07-13 | 平均分：7.4/10 | 主要错误模式：遗漏型（4/5 题）

---

## 一句话定义

Harness Engine 是套在 AI 模型外面的**评估脚手架**：驱动模型跑测试数据集，解析输出，计算多维度评估指标，生成报告，检测退化。它**不是推理引擎**，不负责模型推理性能。

---

## 五大核心概念

| # | 概念 | 一句话 | 你的薄弱度 |
|---|------|--------|-----------|
| 1 | 四层架构 | 输入层 → 模型层 → 输出层 → 评估层，数据单向流动 | ⚠️ 高（每层附加职责遗漏多） |
| 2 | 多维评估 vs 二元判断 | 传统 test harness 用二元判断（输出确定+正确唯一）；AI harness 用多维评估（输出不确定+正确不唯一） | ⚠️ 最高（根本原因没吃透） |
| 3 | CompletionFn 抽象 | 模型调用的抽象接口，让评估逻辑与具体模型**解耦**，切换模型不改评估代码 | ⚠️ 高（解耦价值没记住） |
| 4 | 三种工具的区别 | Inference Engine 关注**性能**；Harness Engine 关注**质量**；Agent Framework 关注**任务完成** | ✅ 掌握 |
| 5 | 声明式任务定义 | OpenAI Evals 用 YAML/JSON 定义评估任务（而非代码），降低门槛 | ⚠️ 中 |

## 四层架构完整职责（⚠️ 你的最大弱项）

> 测试中你只说了每层的核心功能，遗漏了附加功能。这里补全。

| 层 | 核心职责 | 你遗漏的附加职责 | 数据流出口 |
|---|---|---|---|
| **输入层** | 从文件/数据库/API 读取数据集，渲染 prompt 模板 | ① Prompt 模板版本管理 + A/B 测试<br>② 变量注入（字段填入模板）<br>③ Few-shot 组织（固定/随机/相似度/分层） | 渲染后的 prompt 字符串 |
| **模型层** | 模型调用的抽象封装（请求封装、响应解析） | ① 采样参数管理（temperature/top_p/max_tokens）<br>② 限速控制（RateLimiter，防 API 超限）<br>③ 本地模型支持（vLLM/Ollama） | 原始响应 + 元数据（延迟/token） |
| **输出层** | 模型响应内容的读取和解析 | ① 错误捕获（API 错误/格式错误/超时）<br>② 输出验证（检查是否符合预期格式） | 解析后的结构化数据（如 `{"label":"work"}`） |
| **评估层** | 和金标/基线对比评估 | ① 多维度指标计算（准确性/语义/效率/安全）<br>② 报告生成（Markdown/JSON）<br>③ **退化检测**（对比基线，超过阈值则告警） | 评估报告 + 退化检测结果 |

**数据流口诀**：测试用例 →（输入层渲染）→ prompt →（模型层调用）→ 原始响应+元数据 →（输出层解析）→ 结构化数据 →（评估层对比）→ 报告+退化检测

## 常见错误（你在测试中犯过的）

### ❌ 错误 1：四层架构每层只记核心职责，遗漏附加功能

```
你说的：输入层=读取数据+填充prompt
实际还有：模板版本管理、A/B测试、变量注入、Few-shot组织

你说的：评估层=和金标/基线对比
实际还有：多维度指标计算、报告生成、退化检测
```

| 层 | 你遗漏的 | 为什么重要 |
|---|---|---|
| 输入层 | Few-shot 组织 | 决定 prompt 质量，影响模型表现 |
| 模型层 | 采样参数 + 限速 | temperature=0 保证可复现；限速防 API 拒绝 |
| 输出层 | 错误捕获 + 验证 | 格式错误不捕获会污染评估结果 |
| 评估层 | 退化检测 | 没有退化检测的 harness 只是报告工具，不是质量门禁 |

### ❌ 错误 2：没点出二元判断 vs 多维评估的**根本原因**

```
你说的：模型输出受采样参数等影响，不确定 → 不能只用二元判断
实际更根本的原因：即使 temperature=0（输出确定），LLM 的输出也没有唯一正确答案
```

| | 传统 test harness | AI harness |
|---|---|---|
| 输出特性 | **确定性**：给定输入，输出固定 | **不确定**（受采样参数影响）+ **正确不唯一**（多种写法都对） |
| 判断方式 | 期望输出 == 实际输出 | 多维度评估（准确性/语义/效率/安全） |
| 举例 | `add(1,2)` 必须返回 `3` | 摘要任务：同一段对话可以有多种正确摘要 |

### ❌ 错误 3：CompletionFn "绑定特定模型 API" → 应理解为"解耦"

```python
# CompletionFn 是抽象接口，不是绑定特定模型
# 切换模型只需切换 CompletionFn 实现，评估代码不变

class CompletionFn(ABC):
    def __call__(self, prompt: str) -> str: ...

class OpenAIFn(CompletionFn): ...    # GPT-4
class ClaudeFn(CompletionFn): ...    # Claude
class LocalFn(CompletionFn): ...     # 本地模型

# 评估代码不关心用哪个模型
evaluator.run(task, completion_fn=OpenAIFn())   # 测 GPT-4
evaluator.run(task, completion_fn=ClaudeFn())   # 测 Claude，评估逻辑不变
```

| | 没有 CompletionFn 抽象 | 有 CompletionFn 抽象 |
|---|---|---|
| 换模型 | 重新实现一套评估代码 | 只切换 CompletionFn 实现 |
| 代码量 | 越来越臃肿 | 评估逻辑写一次 |
| 测试流程 | 每次换模型都要重新测试流程 | 评估流程不变 |

### ❌ 错误 4：迁移解耦思路时遗漏评估层

```
你说的：输入层和输出层可以解耦
实际遗漏：评估层也可以解耦（而且最重要）
```

| 层 | 可解耦点 | 不同实现 |
|---|---|---|
| 输入层 | 模板渲染抽象 | 分类模板 / 摘要模板 / 翻译模板 |
| 输出层 | 解析器抽象 | 分类标签解析 / JSON 解析 / 自由文本解析 |
| **评估层** | **评估器抽象** | **EM / F1 / BLEU / LLM-Judge，通过配置选择（lm-eval 的 metric_list）** |

## 真实例子

### 例 1：三种工具的选择判断

| 场景 | 选择 | 理由 |
|---|---|---|
| 生产环境实时聊天，1000 QPS，低延迟 | Inference Engine | 关注性能（延迟/吞吐/GPU） |
| 测试 GPT-4 在 100 道数学题上的准确率，对比退化 | Harness Engine | 关注质量（准确性/覆盖度/回归） |
| LLM 自动浏览网页、填表单、完成订票 | Agent Framework | 关注任务完成（工具调用/多步推理） |

### 例 2：OpenAI Evals 架构

```
Eval（一次评估运行）
  ↓ 调用
CompletionFn（模型调用抽象，与具体模型解耦）
  ↓ 被编排
EvalRunner（编排评估流程：加载→调用→解析→评估→报告）
  ↓ 管理
Registry（注册和管理评估任务）
```

关键设计决策：
- CompletionFn 抽象 → 评估与模型解耦
- YAML/JSON 声明式定义任务 → 低代码添加评估任务
- 结果以 JSONL 存储 → 便于后续分析

## 使用前检查清单

设计 harness 前快速过一遍：

- [ ] 是否清楚四层架构每层的**完整职责**（不只是核心功能）？
- [ ] 输入层是否有 Prompt 模板版本管理和 Few-shot 组织？
- [ ] 模型层是否设置了采样参数（temperature=0 保证可复现）和限速？
- [ ] 输出层是否有错误捕获和输出格式验证？
- [ ] 评估层是否包含退化检测（不只是对比金标）？
- [ ] 是否用 CompletionFn 抽象让评估与模型解耦？
- [ ] 是否理解传统二元判断不适用于 LLM（正确不唯一）？
- [ ] 评估层是否支持多指标插件化（EM/F1/BLEU/LLM-Judge 可选）？

## 5 个快速自测问题（30 秒内回答）

1. **Harness Engine 四层架构中，评估层除了"和金标对比"还有哪三个附加职责？**
   > 多维度指标计算、报告生成、退化检测。

2. **为什么传统 test harness 用二元判断，AI harness 用多维评估？请说出两个根本原因。**
   > ① 传统软件输出**确定性**（给定输入输出固定）；② LLM 输出**正确不唯一**（即使 temperature=0，也有多种正确写法）。

3. **CompletionFn 抽象的作用是什么？如果没有它，换模型时会怎样？**
   > 让评估逻辑与具体模型解耦。没有它，每次换模型都要重新实现一套评估代码，代码臃肿，且重新测试流程成为额外变量。

4. **Harness Engine、Inference Engine、Agent Framework 分别关注什么？**
   > Harness 关注**质量**（准确性/覆盖度/回归）；Inference 关注**性能**（延迟/吞吐/GPU）；Agent 关注**任务完成**（工具调用/多步推理）。

5. **harness 四层中，哪一层可以像 CompletionFn 一样做"插件化解耦"？举例说明。**
   > 评估层。不同指标（EM/F1/BLEU/LLM-Judge）可以作为插件注册和选择，新增指标不需要改 harness 核心代码。lm-eval-harness 的 `metric_list` 就是这个思路。
