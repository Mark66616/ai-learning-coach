## 1. 定义、边界与核心概念

### Prompt 模板系统解决什么

Prompt 模板系统是 Harness 输入层的一部分：将**固定指令**、**动态变量**、**few-shot 示例**、**模板版本**从业务调用中分离，用统一、可版本化的方式渲染出最终发送给模型的 Prompt。

它不是“把字符串放进文件”这么简单，而是为了建立以下闭环：

```text
测试用例 + 模板版本 + few-shot 策略
              ↓ 渲染
        最终 Prompt（可保存）
              ↓ 调用
模型版本 + 采样参数 + 随机状态
              ↓
原始输出 → 解析结果 → 逐例指标 / 聚合报告
```

没有模板系统时，Prompt 散落在多处 f-string：同一任务格式逐渐漂移、修改难审查、不同评估运行无法公平对比。模板系统的目标是：**业务代码负责调用，模板负责表达任务，配置负责选择版本，运行记录负责证明结果。**

### 五个必须分清的概念

| 概念 | 它是什么 | 关键使用条件 |
|---|---|---|
| Jinja2 模板 | `.j2` 文件中的文本与控制结构 | `{{ variable }}` 注入数据；`{% if %}` / `{% for %}` 组织可选内容与示例。 |
| 变量校验 | 防止用例数据不完整或类型错误 | `StrictUndefined` 解决模板引用缺失；业务层再校验任务要求的字段与类型。 |
| 模板版本 | Prompt 的可追溯身份 | 不用“最新文件”覆盖旧版；保留 `v1/v2/v2.1`、父版本、变更原因、哈希。 |
| A/B 实验 | 比较两个模板变体的实验设计 | 除模板外的关键变量必须保持一致；结论来自逐例证据和分层指标。 |
| few-shot 选择 | 从标注示例池挑例子拼入 Prompt | 选择策略本身会影响结果，必须配置化、可记录、可复现。 |

### 模板方案何时选

| 方案 | 优点 | 局限 | 适用 |
|---|---|---|---|
| f-string | 零依赖，阅读直接 | 条件/循环弱，Prompt 易散落，难版本化 | 5 个以内的临时简单 Prompt。 |
| Jinja2 | 变量、条件、循环、文件加载成熟 | 需约束模板与数据边界 | 正式 Python Harness 的首选。 |
| Mustache | 逻辑少、跨语言一致 | 复杂组织能力较弱 | 多语言客户端共享模板。 |
| 自定义 DSL | 任务语义完全可控 | 设计、维护与迁移成本高 | 合规、审计或特殊编排要求强的场景。 |

## 2. 你的验收作答：证据、补充概要点与完整解答思路

### 第 1 题：为什么不能把 Prompt 分散写在业务 f-string 中？— 8/10

**你的原答**：

> 分散写在业务代码中，会造成业务代码杂乱（如多个文件同一个 prompt 写多次）、prompt 格式不固定（影响模型结果），并且格式不同那么就是测试中的一个变量，模型结果有差异不可复现。

**你已经答对的部分**：你准确识别了重复维护、格式漂移和“Prompt 也是实验变量”这三个核心点。

**补充概要点**：把“可复现”从口号变成可验证事实：一次运行必须能找回当时的**模板内容**、**最终渲染 Prompt**和**模型调用配置**。

**完整解答思路**：

1. 业务代码中的 f-string 同时混入了“任务规则”“数据填充”“模型调用”。同一个分类规则在三个文件各写一遍时，维护者只改其中一处，就会形成三个近似但不相同的 Prompt；这就是维护性问题。
2. Prompt 的措辞、标签顺序、输出限制、是否提供示例，都会改变模型输出。若同一评估集的用例实际使用了不同格式，结果差异可能来自格式而不是模型能力；因此 Prompt 是必须控制的自变量。
3. 模板文件把固定规则收敛到一个版本化位置，测试用例只传 `email_text` 等数据。这样“改规则”和“换数据”是两种清楚的改动，可以分别审查。
4. 仅记录“我用了 v2”仍不充分：`v2.j2` 可能后来被覆盖，或渲染时 few-shot 发生变化。应记录模板版本与哈希、最终 Prompt、模型快照、采样参数、选例 ID。具备这些证据，才能重跑并解释一次低分。

### 第 2 题：如何把 `v2.1` 高 4% 归因于模板？— 6/10

**你的原答**：

> 首先固定测试数据集、few-shot 的实例数据模型的参数，然后比较提示词模板的差异，一点点调整新版本的提示词模板，多次测试查看差异。

**你已经答对的部分**：你已经在做对照实验的核心动作——固定测试集、few-shot、模型参数，只逐步改模板，并通过多次测试观察差异。

**补充概要点**：实验要把“除模板以外可能影响输出的变量”全部固定或记录；`seed` 是使随机抽样可复现的起点，相同 seed、示例池和抽样规则会得到同一组 few-shot 示例。

**完整解答思路**：

1. 先提出可检验的假设：`v2.1` 的改动会提升某项指标，而不是“看起来更好”。明确成功指标、最低可接受提升和不能退化的指标，例如 spam 准确率不得下降、格式合规率不得下降。
2. 冻结数据：两版本必须跑相同数据集版本、相同用例 ID、相同参考答案。否则 `v2.1` 也许恰好拿到了更容易的题，4% 不是模板效果。
3. 冻结模型调用：模型名称还不够，尽量记录具体快照或版本、系统消息、工具配置、`temperature`、`top_p`、`max_tokens`。这些任何一个改变，都可能改变输出分布。
4. 冻结 few-shot：若固定策略，直接使用相同示例；若随机策略，用独立随机数生成器和 seed。seed 的作用不是“让模型更聪明”，而是让随机选例过程可重放。例如 `seed=42` 使 v2 与 v2.1 使用同一批示例；不固定 seed 时，性能差异可能只是抽到了更好的例子。
5. 保留逐例证据：模板版本/哈希、渲染 Prompt、选例 ID、原始输出、解析结果、正确与否、延迟。没有逐例结果，就无法知道 4% 来自哪些题、哪些类别或是否只是输出格式变化。
6. 处理模型随机性：若模型/API 的 seed 不可控，不能靠单次结果下结论。让两版在同条件下重复 N 次，比较平均值和波动；提升小于正常波动时，结论应是“证据不足”。
7. 最后分层比较：总体准确率之外，看 work/personal/spam、长文本、边界用例和格式合规率。只有在这些控制条件下，才能谨慎表述“提升主要来自模板改动”。

### 第 3 题：`render()` 的主链路是什么？— 7/10

**你的原答**：

> task_name 缺失就要返回异常……variables 可以根据不同版本 prompt 设定需要哪些参数名，然后校验参数及其类型是否正确……version 可以缺失，如果不存在或者为 None 可以读取配置的当前版本的 prompt 配置。

**你已经答对的部分**：任务不能猜默认值；变量应按模板要求校验；调用者没有指定版本时，读取 `current_version` 是合理默认。

**补充概要点**：区分“可安全默认”和“必须失败”。只有 `version=None` 可默认到配置当前版；未知任务、显式但不存在的版本、必填变量缺失和类型不匹配都必须失败。

**完整解答思路**：

1. 用 `task_name` 查配置。它决定任务属于邮件分类还是摘要，也决定模板目录、允许版本、所需字段。找不到任务时不能猜一个默认模板，否则调用者可能得到完全错误的能力。
2. 确定版本：若调用者传 `None`，读取 `current_version`；若传了 `v9` 却不存在，必须报错。对显式请求静默回退到当前版，会掩盖配置错误并让实验失去版本语义。
3. 读取版本元数据和模板文件。例如元数据可说明 `template: v2.j2`、`required_variables: [email_text, labels]`、是否允许 few-shot。
4. 在渲染前校验业务语义：必填字段是否齐全、`labels` 是否为标签列表、few-shot 是否为含 `text/label` 的字典列表。这样错误信息会面向调用者，而不是只给模板异常。
5. 用 Jinja2 `Environment(..., undefined=StrictUndefined)` 加载并渲染。它处理模板内变量遗漏，例如模板引用 `email_text` 但 variables 没有此键时立即失败。
6. 返回字符串前记录任务名、选定版本、模板哈希、最终 Prompt 和必要运行上下文；调用层再把该 Prompt 交给模型。记录的目的不是业务功能，而是让评估可审计和复现。

### 第 4 题：为什么不能只看 A/B 总平均？— 5/10

**你的原答**：

> 没有比较两个版本的实验次数以及分别的准确率，只采用“总体平均准确率”相当于只看总不看分……不同场景的准确率没有做对比，就不能够证明出配置是大多数方面正确率都偏高。

**你已经答对的部分**：总体平均会把不同场景混在一起；必须分别比较场景/类别指标，才能判断优势是否普遍存在。

**补充概要点**：总体平均只回答“全部样本合起来怎样”，不能回答“关键类别有没有退化”“两版本是否在公平样本上比较”“4% 是否超过随机波动”。

**完整解答思路**：

1. 平均值会掩盖少数类。若 work 占 80%，v2.1 把 work 从 90% 提到 95%，却把 spam 从 82% 降到 65%，总体可能上升，但对 spam 用户已明显变坏。
2. 样本量决定可信度。一个版本只有 20 条 spam 样本，另一个有 200 条时，两者平均值的稳定性不同；比较前应确认样本量和流量分配，最好让两版跑同一批用例。
3. 数据分布会混淆结论。v2 只碰到短邮件而 v2.1 碰到长邮件，或相反，差异不能归因给模板；同集配对运行最能消除这个问题。
4. 单次运行有随机波动。随机 few-shot、非零 temperature 或服务端不确定性都可能让结果上下浮动。需要固定 seed，或重复多轮并比较均值、标准差/区间；4% 小于波动范围时不能宣布胜出。
5. 质量不只准确率。分类任务还要看输出是否仅含合法标签；生产系统还可能有延迟、token 成本、安全违例等硬约束。一个平均准确率略高但延迟翻倍的模板，未必值得替换。
6. 因此报告应同时包含：总分、各标签/难例切片、样本量、逐例变化、重复运行波动、格式合规率、延迟和成本，并按业务门槛作发布决策。

### 第 5 题：类别不均衡且主题差异大时如何选 few-shot？— 7/10

**你的原答**：

> “类别极不均衡”会造成不同类别的准确率或者评估结果的差异比较大；“输入主题差异大”无法确定评估结果的差异是数据集还是 prompt 的原因。可以使用相似度 few-shot 解决输入主题差异大的问题，使用分层 fewshot 解决类别及不均衡的问题。

**你已经答对的部分**：相似度解决输入与示例是否相关；分层解决少数类别在 Prompt 中被大类挤掉。这两个策略应组合，而不是二选一。

**补充概要点**：先用分层设置“每类至少出现多少示例”的覆盖约束，再在每一类内部按相似度选例；最后把选择过程作为评估记录的一部分。

**完整解答思路**：

1. 若从 80% work 的池子里直接随机取 3 条，很容易三条都是 work，模型看不到 personal/spam 的判别边界。这不是模型一定不会分类，而是 Prompt 没有提供完整标签覆盖。
2. 先按标签分桶：`work`、`personal`、`spam`。设定最低配额，例如每类至少 1 条；如果某类不足，报错或使用预先定义的补齐方案，不能静默忽略。
3. 分层只解决覆盖，不保证例子贴近当前邮件。例如“客户报价需要今天确认”与“妈妈问周末是否回家”都覆盖了类别，但前者更需要与报价、会议、客户相关的 work 示例。
4. 因此在每个桶内使用 embedding 相似度检索：输入邮件作为 query，选择同类里最相近的例子。总 token 预算不足时先保留每类 1 条，再将剩余额度给全局相似度更高的例子。
5. 为可复现，记录候选池版本、embedding 模型及版本、相似度计算/排序、最终 example ID；如果最后需要随机打破并列或补样，记录 seed。这样下次能知道 Prompt 中到底出现了哪些例子，而不是只知道策略名。

**你的主要薄弱模式**：概念方向正确，但方案需要补齐“控制变量 → 留存证据 → 分层比较 → 按门槛决策”的工程闭环。

## 3. 实现与实验：可直接复用的方案

### 例 1：文件模板与严格渲染

`templates/email_classification/v2.j2`：

```jinja2
你是邮件分类助手。
可选分类：{{ labels | join("、") }}。

{% if few_shot_examples %}
请参考示例：
{% for example in few_shot_examples %}
<example>
邮件：{{ example.text }}
分类：{{ example.label }}
</example>
{% endfor %}
{% endif %}

<email>{{ email_text }}</email>
只输出一个分类名称，不要解释。
```

```python
from pathlib import Path
from jinja2 import Environment, FileSystemLoader, StrictUndefined

templates_dir = Path(__file__).parent / "templates"
env = Environment(
    loader=FileSystemLoader(templates_dir),
    undefined=StrictUndefined,
    trim_blocks=True,
    lstrip_blocks=True,
)
```

`StrictUndefined` 的价值：若漏传 `email_text`，运行在渲染阶段就失败，而不是把 `{{ email_text }}` 或空内容送到模型后才得到难解释的低分。

### 例 2：`PromptEngine.render()` 的可靠实现骨架

配置应提供 `name`、`current_version`、各版本的模板文件、必填变量和变更元数据。以下代码省略 YAML 读取细节，但完整展示关键边界：

```python
from hashlib import sha256
from typing import Any

class PromptEngine:
    def render(
        self, task_name: str, variables: dict[str, Any], version: str | None = None
    ) -> str:
        config = self.configs.get(task_name)
        if config is None:
            raise KeyError(f"未知任务：{task_name}")

        selected_version = version or config["current_version"]
        meta = config["versions"].get(selected_version)
        if meta is None:
            raise KeyError(f"任务 {task_name} 不存在版本：{selected_version}")

        required = meta.get("required_variables", [])
        missing = [name for name in required if name not in variables]
        if missing:
            raise ValueError(f"版本 {selected_version} 缺少变量：{missing}")

        template_path = self.templates_dir / meta["template"]
        template_hash = sha256(template_path.read_bytes()).hexdigest()
        template = self.env.get_template(meta["template"])
        prompt = template.render(**variables)  # StrictUndefined 再做模板级兜底
        self.run_logger.info({
            "task": task_name,
            "template_version": selected_version,
            "template_hash": template_hash,
            "rendered_prompt": prompt,
        })
        return prompt
```

实现时的注意点：

- 不要让外部用户提交模板源码；外部输入只能作为 `variables` 数据。
- `version=None` 可以安全地使用配置默认版；**传入不存在版本**必须失败，不能悄悄回退。
- 变量类型校验可用 dataclass、Pydantic 或自写校验；模板变量与任务元数据必须同名。
- 渲染后的 Prompt 可能含隐私数据；生产日志需要脱敏、访问控制和保留期限，而不是无限期明文存储。

### 例 3：A/B 的可复现运行记录

将“模板版本”当作一次运行的一个字段，不当作结论本身。最小 JSONL 记录示意：

```json
{
  "run_id": "2026-07-13-email-ab-001",
  "case_id": "spam-042",
  "dataset_version": "email_eval_2026-07-13",
  "template_version": "v2.1",
  "template_hash": "...",
  "model": "model-name-and-snapshot",
  "sampling": {"temperature": 0, "top_p": 1, "max_tokens": 8, "seed": 42},
  "few_shot_strategy": "stratified_then_similar",
  "selected_example_ids": ["work-012", "personal-003", "spam-019"],
  "raw_output": "spam",
  "parsed_label": "spam",
  "reference_label": "spam",
  "correct": true,
  "latency_ms": 412
}
```

比较 `v2` 与 `v2.1` 的正确流程：

1. 冻结测试集版本及用例 ID；两版跑同一批用例。
2. 固定模型快照、系统消息、工具配置、采样参数和 few-shot 选择策略。
3. 若策略或模型仍随机，固定 seed；若 API 不保证 seed，则做多次独立重复并报告波动。
4. 保存每例证据，计算总体指标、各标签指标、难例指标、格式合规率、延迟和成本。
5. 检查回归：`v2.1` 总分更高但 spam 明显下降，或成本/延迟显著变差，都不能简单宣布替换。
6. 结论写成“在数据集 X、模型配置 Y、运行 N 次下，v2.1 的平均指标为……”，不要泛化成“v2.1 永远更好”。

### 例 4：分层 + 相似度 few-shot

```text
示例池按标签分桶
  work:     [...]
  personal: [...]
  spam:     [...]
      ↓
每类保留至少 1 个名额（覆盖约束）
      ↓
在各桶中按 query 与示例的 embedding 相似度排序（相关性）
      ↓
按总 token 预算截断，记录最终 example_id 列表
```

- **固定**：固定三条例子，最适合建立稳定基线和排查 Prompt 改动。
- **随机**：探索示例池质量或减少固定示例偏差；使用独立 RNG，记录 `seed`，避免全局随机状态。
- **相似度**：主题、意图跨度大时优先；记录 embedding 模型、向量库/候选池版本、相似度和排序。
- **分层**：标签不均衡或类别覆盖为硬要求时优先；某类示例不足时应报错或按明确策略补齐，不能静默偏向大类。

## 4. 常见错误、诊断与交付前检查

### 常见错误 → 为什么错 → 如何修

| 错误 | 为什么会出问题 | 修正方式 |
|---|---|---|
| Prompt 散落在多个 f-string | 同任务格式不一致，改动难发现，评估不再可比 | `.j2` 文件 + 配置选择版本 + 代码审查模板 diff。 |
| 缺变量时渲染空文本 | 模型失败原因被误判为模型/Prompt 质量 | `StrictUndefined`、必填字段/类型校验、渲染前测试。 |
| “随机 A/B”只跑一次 | 结果可能来自抽样、流量或模型随机性 | 同集配对运行；固定 seed 或重复运行并报告方差。 |
| 只比较总体准确率 | 少数类、难例、格式错误与安全退化被平均值吞没 | 分层指标、逐例 diff、合规率、延迟/成本共同决策。 |
| 将 `current_version` 当成“唯一真相” | 旧实验无法还原，默认切换影响历史结果 | 每次运行记录显式版本和模板哈希。 |
| 相似度检索不记录候选池 | 选例变化无法解释，结果不能复现 | 冻结候选池版本，记录检索器、排序与最终例子。 |
| 用户输入作为模板执行 | 用户可影响模板控制结构，扩大风险面 | 模板可信且受版本控制；用户输入仅作为 render 参数。 |

## 6. 参考答案与复习行动

### 1. 模板系统的三种价值

- **维护性**：模板文件集中管理，业务代码只调用 `render()`；改动可通过模板 diff 和版本元数据审查。
- **可比性**：同一模板/版本、同一渲染规则让用例间格式一致；Prompt 格式不再是隐藏变量。
- **可复现性**：保存版本/哈希、最终 Prompt、模型/采样参数、few-shot 选例与用例 ID，可重跑同一实验。

### 2. `render()` 正确流程

校验 `task_name` → 查任务配置（未知任务报错）→ `version` 为空时取 `current_version` → 查版本元数据（未知显式版本报错）→ 校验必填变量和类型 → 加载 `.j2` 模板 → 用 `StrictUndefined` 渲染 → 记录任务、版本、哈希、最终 Prompt → 返回字符串。  
可默认：`version=None`。不可默认：未知任务、未知显式版本、必填变量缺失、类型不匹配。

### 3. A/B 归因清单

至少控制/记录：数据集版本、用例 ID、模板版本/哈希、最终 Prompt、模型快照、系统消息、采样参数、seed、few-shot 策略与选例、工具配置、运行次数、原始输出、解析结果、逐例指标。  
除总体准确率外，看标签/难例切片、样本量、置信或波动、格式合规率、延迟、成本与安全指标。只有“模板不同，其余证据可追溯”时，提升才可主要归因于模板。

### 4. 不均衡且主题差异大的选择器

为每类保留最低名额（如每类 1 条）以防 80% 的 work 挤掉少数类；在每类中按 embedding 相似度排序选最相关示例；总数/字符数受 token 预算限制时，先满足覆盖再按全局相似度补齐。记录候选池、检索模型/版本、相似度、排序、最终 ID；随机补样使用固定 seed。示例池不足应报错或按预先声明的规则补齐。

### 5. 总分上升但关键指标退化时的决策

报告应同时展示总体、spam、其他类别、延迟、成本及样本量；标记 `v2.1` 在 spam 与延迟上的回归。是否发布取决于业务权重与预设门槛，例如 spam 是安全关键类别时应阻断发布；可保留 `v2` 为默认，针对性修复 spam 模板/示例后重跑。总平均无法抵消关键类别或服务质量的显著退化。
